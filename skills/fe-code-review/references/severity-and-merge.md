# Severity And Merge

`fe-code-review` 的价值不只是列问题，还要把问题分级、判断阻塞性，并给出版本级合并建议。

## 严重级别定义

### `P0`

合并前必须修复。

典型特征：

- 确定性功能断流
- 明显运行时异常
- 错误提交或明显数据丢失
- 核心主链路无法完成

### `P1`

高优先级，不应直接合并。

典型特征：

- 明显回归风险
- 核心链路状态不一致
- 关键切换场景下容易进入错误状态
- 跨模块联动后留下高概率缺陷

### `P2`

中优先级，通常存在绕行方式。

典型特征：

- 用户可感知缺陷
- 数据不一致但不一定立即致命
- 稳定性、扩展性、复用性上的明显问题
- 某些边界链路需要额外回归或后续修正

### `P3`

低优先级，非阻塞问题。

典型特征：

- 体验问题
- 边界文案或显示瑕疵
- 不影响当前版本是否能合的质量债

## 置信度定义

### `高`

- 代码路径可直接闭环
- 问题触发链条清晰
- 不依赖额外环境假设就能成立

### `中`

- 推断较强
- 依赖部分运行态、接口行为或上下文前提
- 结论大概率成立，但不是完全闭环

### `低`

- 更偏结构性风险、演进性隐患或需额外验证的怀疑点
- 适合提醒，不适合装作确定性问题

## 阻塞性判断

默认规则：

- `P0` 一律阻塞
- `P1` 默认阻塞
- `P2` 需要结合业务链路判断是否阻塞
- `P3` 默认非阻塞

如果一个问题会直接影响“是否建议合并”的结论，就应标记为阻塞。

## 四档合并建议

### `建议合并`

适用条件：

- 无 `P0`
- 无 `P1`
- 无阻塞性 `P2`

### `可合并，但建议后续修复/优化`

适用条件：

- 只有非阻塞 `P2 / P3`
- 不影响核心业务路径正确性
- 当前版本可以带着这些问题推进，但应明确记账

### `修复关键问题后合并`

适用条件：

- 存在 `P1`
- 或存在阻塞性 `P2`
- 但整体还没有糟到“明显不应合并”

### `暂不建议合并`

适用条件：

- 存在 `P0`
- 或多个核心路径 `P1` 叠加
- 或主流程状态一致性明显失稳

## 输出语气

分级和合并建议都应该基于技术事实，不要写成情绪判断。

推荐语气：

- “当前存在 X 类阻塞问题，不建议直接合并”
- “当前问题主要集中在状态迁移和回归风险，建议先修复关键问题后再合并”
- “当前没有阻塞项，但建议后续补上这些优化”

避免写法：

- “代码质量太差”
- “这版看起来不行”
- “感觉风险挺高”

结论要能让团队直接拿去沟通，而不是再翻译一遍。
