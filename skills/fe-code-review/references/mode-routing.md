# Mode Routing

`fe-code-review` 对外是统一入口，对内按 review 场景分模式。

固定心智是：

> 只要是前端 review，请先进入 `fe-code-review`，再决定具体模式。

V1 只正式支持 `version-review`，但这个 skill 仍然要先接住所有前端 review 请求；路由目标不是“只接自己会完整处理的请求”，而是“统一入口接住请求，再把未支持模式识别出来并正确追问”。

补充边界：

- V1 的触发重点不是覆盖所有模糊说法
- 重点是用户已经明确表达“要 review / 审查 / 代码审查”
- 如果用户根本没有 review 意图，只是在问实现方式、排查 bug、聊方案，不要仅凭文件路径硬触发

## V1 模式表

| 模式 | 典型输入 | V1 是否支持 | 处理方式 |
| --- | --- | --- | --- |
| `version-review` | 审分支、PR、版本实现、相对 `master` / 主干的改动、能否合并、整体影响、回归风险 | 支持 | 直接进入版本级审查 |
| `focused-review` | 指定少量文件或单个模块，希望只看局部实现 | 未正式支持 | 说明是后续扩展位，追问是否改按版本级审查 |
| `staged-review` | 只看当前暂存区、当前小范围改动 | 未正式支持 | 说明是后续扩展位，追问是否改按版本级审查 |
| `re-review` | “我按你上轮意见修了一版，再帮我看” | 未正式支持 | 说明是后续扩展位，追问是否改按版本级审查 |

## 直接进入 `version-review` 的信号

在用户明确提出 review 意图的前提下，出现下列任意组合，都应优先路由到 `version-review`：

- 分支名、PR、commit range、仓库路径
- “整体影响”“回归风险”“能否合并”“版本审查”
- “相对 master 看一下”“帮我看这次版本实现”
- “先给结论，再看详细问题”

## 不要误判成 `version-review` 的请求

这些请求在 V1 不要直接硬套：

- “帮我 review 这两个文件”
- “只看一下 staged changes 有没有明显问题”
- “我修过一轮了，帮我做复审”
- “看看这个组件写法有没有问题”

正确处理方式：

1. 明确说当前 V1 重点支持版本级首轮审查
2. 说明该请求更像 `focused-review` / `staged-review` / `re-review`
3. 追问用户是否改按版本级审查处理

## 路由不清时的最小追问

只问必要信息，不要把用户拖进长问卷。

优先补这两类信息：

1. 本次 review 的对象是什么
   - 整个分支 / PR / 一段 diff
   - 还是几个文件 / staged changes
2. 用户要的结果是什么
   - 版本级结论、合并建议、回归风险
   - 还是局部实现问题

## 基线规则

- `version-review` 默认按远端最新 `origin/master`
- 如果用户明确指定其他基线，按用户输入处理
- 如果远端基线不可读，先说明，再让用户确认下一步
- 不要静默 fallback 到本地 `master`

## 一个稳定心法

先问“用户是不是在明确要求做 review”，再问：

> 他到底是在要一个版本级结论，还是只要局部代码意见？

这两个问题回答清楚，路由通常就清楚了。
