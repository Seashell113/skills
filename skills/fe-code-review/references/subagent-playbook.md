# Subagent Playbook

V1 默认单 agent 完成版本级审查。

子 agent 是亮点能力，但它的价值来自“拆得对”，不是“并行得多”。因此拆分规则必须稳定，不能变成平均切 diff。

## 启用前提

必须同时满足：

1. 用户明确允许并行拆分
2. 主 agent 的预扫描命中任一门槛：
   - 变更文件数 `>25`
   - `added + deleted > 800`
   - 能识别出 `>=3` 个热点模块
   - 存在 `>=2` 条相对独立的业务路径

如果只满足门槛但用户没授权：

- 保持单 agent 流程
- 可以在摘要中说明“这次改动较大，如需提高覆盖面，可授权并行拆分”

## Review Map

在派发任何子 agent 之前，主 agent 必须先形成 `review map`。

固定字段：

- 基线
- diff 概览
- 热点模块
- 候选业务路径
- 核心风险假设
- 推荐拆分方式

没有 `review map`，不允许直接派发子 agent。

## 拆分策略

### 第一优先级：按业务/模块范围拆分

当改动可以稳定分成 `2-4` 个相对独立区域时，优先按模块或业务路径拆。

适用信号：

- 每个区域都具备相对完整的业务上下文
- 区域之间交叉较少
- 每个区域至少覆盖 `3` 个文件或约 `20%` 的改动量

示例：

- 列表与筛选
- 详情表单
- 提交流程
- 公共搜索组件

### 第二优先级：按审查角度拆分

当改动高度集中在一个大模块或一条长链路上，无法按模块无重叠拆分时，再按关注点拆。

固定角度：

1. 功能正确性 / 状态流 / 逻辑漏洞
2. 影响面 / 回归风险 / 兼容性
3. 稳定性 / 可扩展性 / 复用性

按角度拆分是兜底方案，不是默认首选。

### 按量只作辅助

按量只能用于：

- 判断“是否该拆”
- 拆分后的工作量均衡

禁止把拆分设计成：

- 前 N 个文件给 agent1
- 后 N 个文件给 agent2

这种拆法会打断上下文，并放大重复发现和漏审的概率。

## 子 Agent 数量

- 上限 `4`
- 推荐 `2-3`
- 主 agent 永远保留协调权和最终判断权

## 子 Agent 输入契约

每个子 agent 至少要拿到这些信息：

- 审查目标
- 负责范围
- 排除范围
- 基线
- 必须关注的问题类型
- 输出字段
- 最多提交 `5` 条候选 finding

如果是范围拆分，要清楚列出负责模块和排除模块。

如果是角度拆分，要清楚列出自己负责的关注点，避免多个子 agent 审同一维度。

## 子 Agent 输出契约

固定输出结构：

- 已审范围
- 未审范围
- 候选 findings
- 需要人工确认的不确定点
- 建议回归路径

额外要求：

- 不要输出纯风格、格式、命名类 checklist
- 不要把低置信度怀疑点伪装成高确定性问题
- 不要直接给最终合并结论，合并结论由主 agent 统一判断

## 汇总规则

主 agent 汇总时固定做这些事：

- 相同根因导致的多处症状，合并成一条 finding
- 不同根因但症状相似，保留为多条
- 严重级别冲突时，取更高一级
- 置信度冲突时，取更保守的一档
- 统一改写成中文口径，不直接拼接子 agent 原文

## 失败兜底

如果任一子 agent：

- 跑偏
- 重复度高
- 输出质量差
- 结论不可靠

主 agent 可以直接丢弃其结果，并继续完成总报告。

如果并行结果不足以支撑最终结论：

- 回退到单 agent 汇总
- 不要为了维持并行的表面形式而输出低质量结论
