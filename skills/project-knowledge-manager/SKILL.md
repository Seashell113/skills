# 项目知识管理

本 skill 用来帮助团队项目把长期有效的信息放到正确位置。目标不是创建一棵看起来完整的目录树，而是让每类重要信息都有唯一、可信、可维护的归属，方便人和 Agent 安全协作。

## 输出语言

默认使用中文输出。除非用户明确要求英文，所有计划、表格、文档草稿、审计报告、命令说明和写入确认都使用中文。路径、命令、文件名、代码标识保持原文。

## 核心原则

先判断信息生命周期，再决定放在哪个文件：

- `README.md`：人如何进入、运行和理解项目入口。
- `AGENTS.md`：Agent 如何安全修改这个具体项目。
- `CLAUDE.md`：Claude Code 的薄适配层，默认只写 `@AGENTS.md`。
- `openspec/`：系统行为、业务能力、需求规格和验收标准。
- `docs/`：非规格类长期辅助知识。
- `.agents/`：项目本地 Agent 扩展材料、规则、流程、模板、脚本和适配。
- 模块 `README.md`：靠近代码的模块长期知识。
- 模块 `AGENTS.md`：只放高风险模块的局部 Agent 约束。

需要做初始化、归位或审计判断时，先读取 `references/directory-model-v1.1.md`。需要起草文件时，读取 `references/templates.md`。用户提到斜杠命令、命令分发或 hook 时，读取 `references/commands.md`。

## 默认安全策略

默认采用“先计划、确认后写入”。

## 来源边界

初始化或更新项目文档时，只沉淀目标项目已有文件、用户明确要求写入的项目事实，以及从代码、配置、文档或命令输出验证出的项目特异信息。

当前会话中的 system / developer 指令、全局 `AGENTS.md`、用户长期偏好、团队通用 rules、工具级行为准则只能指导本次执行，不得默认复制进目标项目。除非用户明确要求在项目内落盘，否则项目 `AGENTS.md` 只写这个项目特有的修改约束、风险路径、验证要求和局部工作流。

如果确实需要在项目内继承团队规则，只写简短引用或说明，不展开复制规则原文。输出初始化计划时，必须显式检查“是否包含上层/全局规则复制”。

最小 `AGENTS.md` 也应包含 `.agents/` 索引位，用来说明项目本地 Agent 扩展材料的统一入口。没有实际内容时不创建 `.agents/` 空目录，但 `AGENTS.md` 要固定说明：若 `.agents/README.md` 存在，应先阅读该索引。

当用户要求初始化、规范化、迁移或沉淀项目文档时：

1. 先检查目标项目现状。
2. 输出写入计划，列出精确路径、职责、创建/更新原因和草稿摘要。
3. 明确列出暂不创建的目录及理由。
4. 明确说明是否包含上层/全局规则复制；默认应为“否”。
5. 等用户明确确认后，再创建或修改目标项目文件。

如果用户在同一条消息中明确要求直接修改，也要先快速检查现状，并保持变更最小。

## /pkm 短触发词定位

`/pkm:*` 适合作为团队内高识别度的短触发词，不适合承载完整知识判断。`/pkm-*` 适合作为不支持冒号命令时的横线命令兼容入口。Claude Code 中优先让用户输入完整 skill 名称 `project-knowledge-manager` 或“项目知识管理 skill”。

推荐短触发词：

- `/pkm:init`：触发标准项目知识初始化规划。
- `/pkm:collect`：回顾当前对话或指定文本，提取需要落盘的信息，给出归位表和写入计划。
- `/pkm:audit`：审计当前项目已沉淀信息的标准性、准确度和唯一真源风险。
- `/pkm-init`、`/pkm-collect`、`/pkm-audit`：横线命令兼容入口，语义分别等同于上面三个短触发词。

这些短触发词默认按自然语言识别处理：看到 `/pkm:init`、`/pkm:collect`、`/pkm:audit`、`/pkm-init`、`/pkm-collect`、`/pkm-audit` 时，按对应工作流执行。不要假设安装本 skill 后，各工具都会自动注册这些斜杠命令。若工具拦截未知斜杠命令或触发不稳定，让用户显式指定 `project-knowledge-manager` 或“项目知识管理 skill”。

命令模板见 `assets/commands/`，说明见 `references/commands.md`。这些模板是可选适配材料，不要求每个项目初始化 `.agents/commands/`。只有项目确实需要共享本地命令或工具适配时，才创建项目内命令文件。

## 工作流 A：初始化或规范化项目

1. 定位项目根目录。未指定路径时，默认使用当前工作目录。
2. 读取现有入口：`README.md`、`AGENTS.md`、`CLAUDE.md`、`openspec/`、`docs/`、`.agents/`、重要模块 `README.md`。
3. 修改前检查 git 状态，保留用户已有改动。
4. 判断信息密度：
   - 小型或早期项目：只保留 `README.md`、`AGENTS.md`、`CLAUDE.md` 和现有源码。
   - 成长期项目：只新增确有内容的 `docs/` 或模块文档。
   - 规格驱动项目：行为规格和变更进入 `openspec/`。
   - Agent 协作密集项目：只有出现项目本地规则、技能、模板、脚本、命令、hook 或适配需求时，才创建 `.agents/`；但最小 `AGENTS.md` 固定保留 `.agents/` 索引说明。
5. 输出写入计划：
   - 路径
   - 用途
   - 创建 / 更新 / 保持不变
   - 原因
   - 草稿摘要
   - 是否包含上层 / 全局规则复制
6. 用户确认后，只创建或更新计划内文件，不创建空目录。

## 工作流 B：从对话或项目资料提取信息

适用于用户说“把刚才讨论沉淀一下”“整理到项目文档里”“把这段对话变成项目知识”“把这些注意事项放到合适位置”“/pkm:collect”等场景。

1. 把输入拆成四类：
   - 已验证事实
   - 判断或设计原因
   - 待决问题
   - 行动项
2. 每条信息只选择一个主归属：
   - 运行命令、启动方式、技术栈、项目入口 -> `README.md`
   - 项目特异 Agent 约束和高风险区 -> `AGENTS.md`
   - 系统行为、业务能力、需求、验收标准 -> `openspec/`
   - 背景、历史、规范、操作手册、ADR -> `docs/`
   - 项目本地 Agent 工作流、可复用检查清单、模板、脚本 -> `.agents/`
   - 模块职责、数据流、权限、路由、埋点、设计包袱、常见坑 -> 模块 `README.md`
   - 高风险模块专属 Agent 约束 -> 模块 `AGENTS.md`
3. 未验证内容不能写成事实，标注“待验证”和来源。
4. 不双写同一事实。其他位置只链接或指向主真源。
5. 更新已有文档时，沿用原文档风格和标题结构。
6. 不把当前会话的上层指令、全局规则或用户长期偏好当作项目事实写入。

## 工作流 C：审计既有知识沉淀

有本地文件系统权限时，先运行本 skill 自带的结构检查脚本。不要假设目标项目根目录存在 `scripts/project_knowledge_audit.py`；应从 skill 安装目录解析脚本路径：

```bash
python3 <skill-dir>/scripts/project_knowledge_audit.py /path/to/project
```

如果运行环境无法定位 skill 安装目录或无法执行脚本，跳过脚本并继续做人工/Agent 语义审计。脚本只能发现结构和启发式问题；仍需结合代码、脚本、CI 和近期事故做准确性审查。

审计报告使用：

```md
# 项目知识沉淀审计

## 摘要
- 总体状态：
- 最高风险问题：
- 推荐下一步：

## 问题清单
| 严重程度 | 路径 | 问题 | 证据 | 修复建议 |
|---|---|---|---|---|

## 信息归位复核
| 信息 | 当前位置 | 更合适的主真源 | 原因 |
|---|---|---|---|

## 待验证或疑似过期内容
| 路径 | 说法 | 不确定原因 | 验证方式 |
|---|---|---|---|
```

严重程度：

- `P0`：正在误导实现、发布、数据、安全、支付、权限或生成代码等高风险工作。
- `P1`：主真源错误、明显双写、项目规则过期，或 Agent 指令可能导致大范围误改。
- `P2`：职责不清、README/AGENTS 过载、导航缺失、信息密度已高但尚未拆分。
- `P3`：命名、格式、风格或小型可维护性问题。

## 需要主动识别的反模式

- 把 `AGENTS.md` 写成项目百科。
- 在 `AGENTS.md` 里重复 README 命令或团队通用规则。
- `CLAUDE.md` 维护另一套项目规则，而不是薄引用 `AGENTS.md`。
- 为了完整性创建空 `docs/`、`.agents/` 或 `.agents/skills/`。
- 把 OpenSpec 管理的业务规格复制到 `docs/`。
- 把团队通用 rules / skills 复制到每个项目。
- 把 system / developer 指令、全局 `AGENTS.md` 或用户长期偏好复制到项目文档。
- 工具专用规则目录成为主真源。
- rules 写成长篇教程。
- skills 只承载单条规则。
- 模块 README 复制类型签名、字段列表或参数细节。
- 模块知识全部抽离到根目录。
- 项目级规范散落在各模块目录。

## 输出要求

使用中文。事实、判断、建议分开写。优先给出具体路径、证据和短表格。不能确认的内容写入“待验证”或“待决问题”，不要伪造确定性。
