# /pkm 短触发词与命令模板

`/pkm:*` 适合做高识别度的短触发词和提示词复用入口，skill 负责真正的判断和执行策略。`/pkm-*` 是不支持冒号命令时的横线命令兼容入口。

本文件描述的是可识别触发词和可选命令模板，不承诺安装 skill 后，各工具都会自动注册 `/pkm:*` 或 `/pkm-*` 命令。团队成员如果发现某个工具拦截未知斜杠命令或触发不稳定，应显式指定 `project-knowledge-manager` 或“项目知识管理 skill”。

## 是否适合

适合。原因：

- 知识收集、初始化和审计都是重复任务，触发语义相对稳定。
- `/pkm:*` 比普通词组更不容易误触发，适合作为团队内约定短写。
- 短触发词可以降低使用门槛，让团队成员不用每次手写完整提示词。
- 短触发词天然适合做“只产出计划，等待确认”的入口，能降低误写风险。
- 后续要接 hook 时，命令提示词可以复用为 hook 产物的人工确认模板。

需要控制的风险：

- `/pkm:*` 默认按自然语言识别处理，不等同于跨工具注册成功的 slash command。
- `/pkm-*` 只有在目标工具中真实注册对应 command 文件，或该 agent 支持文本触发识别时才稳定可用。
- Claude Code 中默认使用完整 skill 名称 `project-knowledge-manager`。
- 不推荐把 `pkm init`、`pkm collect`、`pkm audit` 作为主触发词，避免普通词组误触发。
- 触发后不要直接落盘，除非用户在命令后明确加 `--apply` 或再次确认。
- 触发后不要把所有对话都当作事实，应区分事实、判断、待验证和行动项。
- 命令模板不要替代 skill；模板正文保持薄，只唤起本 skill。
- hook 不宜自动写文档，最多自动提示“可能有可沉淀信息”或生成候选归位表。

## 推荐短触发词

| 触发词 | 目的 | 默认行为 |
|---|---|---|
| `/pkm:init` | 标准项目知识初始化 | 检查项目入口，输出初始化计划，不直接写文件 |
| `/pkm:collect` | 从当前对话或指定文本提取可沉淀信息 | 输出信息归位表和写入计划，不直接写文件 |
| `/pkm:audit` | 审计已沉淀信息 | 运行结构检查并补语义审计，输出问题清单 |

横线兼容命令：

| 触发词 | 等价语义 | 使用场景 |
|---|---|---|
| `/pkm-init` | `/pkm:init` | 目标工具不支持冒号命令，或只支持文件名式 command |
| `/pkm-collect` | `/pkm:collect` | 目标工具不支持冒号命令，或只支持文件名式 command |
| `/pkm-audit` | `/pkm:audit` | 目标工具不支持冒号命令，或只支持文件名式 command |

可选扩展：

| 触发词 | 目的 | 使用条件 |
|---|---|---|
| `/pkm:apply` | 执行上一轮已确认的写入计划 | 只在当前对话存在清晰、刚确认的写入计划时使用 |
| `/pkm:review` | 复核某次写入后的标准性和准确度 | 用于落盘后自检 |

## 全局分发策略

团队全局分发时，推荐只安装 `project-knowledge-manager` skill，不默认在每个项目初始化 `.agents/commands/` 或工具专用 command 文件。

优先使用：

- `/pkm:init`
- `/pkm:collect`
- `/pkm:audit`

Claude Code 中优先使用：

- `project-knowledge-manager`

其他 agent 可使用：

- `/pkm:init`
- `/pkm:collect`
- `/pkm:audit`
- `/pkm-init`
- `/pkm-collect`
- `/pkm-audit`

如果这些短触发词在某个工具里被拦截或识别不稳定，使用显式 skill 触发：

- `project-knowledge-manager`
- `项目知识管理 skill`
- `$project-knowledge-manager`（支持 `$` mention 的工具）

## 命令模板适配

`assets/commands/` 下的模板用于需要真实注册命令的工具或项目本地适配。只有在目标工具明确支持自定义命令时，才把模板复制或转换到工具识别目录。

如果目标工具不支持冒号命令名，使用文件名式适配名：

- `/pkm-init`
- `/pkm-collect`
- `/pkm-audit`

命令正文仍声明用户可见短触发词 `/pkm:init`、`/pkm:collect`、`/pkm:audit`，避免不同工具之间的语义漂移。

## Hook 建议

hook 可以分阶段引入：

1. 第一阶段：不做 hook，只提供手动命令。
2. 第二阶段：在会话结束、提交前或 PR 前做轻量提醒，例如“本次对话可能有 3 条可沉淀信息，是否运行 `/pkm:collect`？”
3. 第三阶段：hook 生成候选归位表，但不自动落盘。
4. 不建议默认启用自动写入文档的 hook。

成本控制：

- hook 输入应只包含最近摘要、改动文件列表和用户明确标记的片段。
- 避免每轮对话都全量扫描历史。
- 对低风险项目可只做提交前审计；高风险项目再考虑更频繁提醒。
