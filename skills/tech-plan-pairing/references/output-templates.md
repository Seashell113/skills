## 核心原则：按受众分层

不同人需要看不同的东西。一份巨大的全包文档不如几份各有侧重的短文档。

| 受众 | 关心什么 | 不需要什么 |
|------|---------|-----------|
| 决策者（技术负责人、管理层） | 为什么做、做什么、代价、收益、风险 | 具体命令、配置文件内容 |
| 执行者（项目负责人、开发者） | 怎么做、步骤、命令、常见问题 | 方案论证过程 |
| 全体成员 | 简洁的操作步骤、验证方式 | 背景分析、技术评估、方案对比 |

**关键判断标准：** 如果一份文档的读者需要跳过一半以上的内容才能找到和自己相关的部分，说明这份文档应该拆分。

## 通用文档层级

根据方案复杂度，从以下层级中选取需要的组合。简单方案可能只需要 2 层，复杂方案可能需要全部 5 层。

### 第 1 层：整体说明（面向决策者）

**什么时候需要：** 方案涉及团队级决策、需要管理层审批或全员同步时。

**核心内容：**

- 要解决什么问题（痛点 + 不做会怎样）
- 团队/项目背景（用数据说话，不要空泛描述）
- 核心决策及理由
- 明确的排除项（"不做什么"及原因）
- 预期收益与风险
- 治理机制（如何跟踪落地进度）

**写作要点：**

- 结论前置，先说"做什么"再说"为什么"
- 数据支撑，避免空泛的定性描述
- 排除项单独成节——"为什么不做 X"能防止后来者重走弯路
- 治理机制要务实：不需要搭建看板的事就别搭看板

**必含内容检查点：**

- [ ] 清楚说明了要解决什么问题（痛点 + 不做的后果）
- [ ] 每个核心决策都有理由
- [ ] 至少列出了"不做什么"（排除项）
- [ ] 预期收益足够让决策者判断投入产出

### 第 2 层：执行方案（面向执行者）

**什么时候需要：** 方案有多个步骤、分阶段推进、或需要不同角色协作时。

**核心内容：**

- 标准清单（版本号、配置值等关键参数，集中在一处）
- 分阶段步骤及每阶段的验证标准
- 特殊场景处理（例外项、边界情况）
- 推进节奏与优先级
- FAQ（从讨论过程中提炼的高频问题，不是凭空假设的）

**写作要点：**

- 命令可直接复制执行
- 每个步骤后有验证方式——执行者需要知道"我做对了吗"
- FAQ 来源于讨论过程中实际碰到的问题，把讨论中的关键问答沉淀下来

**必含内容检查点：**

- [ ] 标准清单集中在一处呈现（不散落在文档各处）
- [ ] 每个阶段有明确的验证标准
- [ ] 特殊/例外场景有处理说明
- [ ] FAQ 来源于真实讨论，非虚构

### 第 3 层：现状盘点（面向项目负责人）

**什么时候需要：** 方案需要对多个项目/系统/模块逐个改造时。

**核心内容：**

- 盘点数据概览（来源、时间、覆盖范围）
- 分类与分布统计
- 每个项目/模块的当前状态与建议动作
- 优先级分类
- 例外清单（无法短期对齐的项目，需包含负责人、到期时间、退出条件）

**写作要点：**

- 数据来源和扫描时间必须明确标注
- 严格区分"现状描述"和"目标建议"——现状保持事实准确，建议指向目标
- 例外不等于不管——例外清单是治理手段，不是遗忘列表

**必含内容检查点：**

- [ ] 数据来源和扫描时间标注明确
- [ ] 现状描述与目标建议严格区分
- [ ] 例外清单每项都有负责人和退出条件

### 第 4 层：技术评估（面向技术决策参考）

**什么时候需要：** 某个关键决策需要深度技术分析支撑，或者有一个被排除的主流方案需要详细解释排除原因时。

**核心内容：**

- 被评估的技术/方案是什么
- 它解决什么问题、吸引力在哪
- 为什么在当前场景下不采用（带具体依据和来源）
- 替代方案是什么
- 什么条件下重新评估

**写作要点：**

- 结论明确（采用/不采用/条件性采用）
- 依据可追溯（引用官方文档、issue、changelog）
- 客观分析为什么不适合当前场景——不要情绪化否定
- "什么条件下重新评估"很重要——技术生态会变，今天不合适不代表永远不合适

**必含内容检查点：**

- [ ] 结论明确（采用/不采用/条件性采用）
- [ ] 关键依据可追溯到来源
- [ ] 写明了"什么条件下重新评估"

### 第 5 层：操作指引（面向全体成员）

**什么时候需要：** 需要全员执行统一操作时。

**核心内容：**

- 标准参数表（一目了然）
- 逐步操作命令（分操作系统/环境给出）
- 验证方式
- 注意事项
- 遇到问题找谁

**写作要点：**

- **只有操作步骤**，没有背景分析、没有方案论证
- 命令可直接复制执行
- 假设读者不了解方案背景——他们只需要知道"做什么"和"怎么验证做对了"
- 如果涉及多个操作系统或环境，每个都要给出具体命令，不要只说"参考官方文档"

**必含内容检查点：**

- [ ] 无背景论证内容，纯操作步骤
- [ ] 命令可直接复制执行
- [ ] 每步有验证方式（执行后如何确认成功）
- [ ] 多平台/环境场景都覆盖到

## 文档组合建议

| 方案复杂度 | 典型场景 | 建议组合 |
|-----------|---------|---------|
| 轻量 | 单一决策，影响面小 | 整体说明 + 操作指引 |
| 中等 | 多个决策，需分步执行 | 整体说明 + 执行方案 + 操作指引 |
| 重型 | 涉及多项目改造、有排除方案需论证 | 全部 5 层按需组合 |

**不要为了形式完整而凑文档。** 如果方案简单到一份文档能讲清楚，就写一份。文档层数不是质量指标。

**也不要把所有东西塞进一份文档。** 如果你发现一份文档超过了 200 行，而且有明显不同的受众段落，考虑拆分。

## 落地质量检查清单

文档全部写完后，执行以下检查。建议用 grep 或全文搜索而非肉眼扫描。

### 一致性检查

- [ ] 同一决策在不同文档中的表述是否一致
- [ ] 关键参数（版本号、工具名、配置值）各文档是否统一
- [ ] 策略描述在概览文档和执行方案中是否口径一致
- [ ] 命令中的参数与标准清单是否匹配

### 内容完整性

- [ ] 每个核心决策是否都有理由说明
- [ ] 排除项是否都有排除原因
- [ ] 例外清单是否都有负责人和退出条件
- [ ] FAQ 是否覆盖了讨论过程中的关键问题

### 可执行性

- [ ] 操作指引中的命令是否可以直接复制执行
- [ ] 每个操作步骤后是否有验证方式
- [ ] 多操作系统/环境场景是否都覆盖到

### 格式规范

- [ ] 章节编号是否有重复或跳跃
- [ ] 表格数据是否对齐、无遗漏列
- [ ] 注释和说明中是否残留了旧版策略的措辞
