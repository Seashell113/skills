# 上下文更新协议

> 当技术方案讨论进入文档落地阶段，并且项目存在长期上下文目录（如 `docs/context/`）时，参考本文件。

---

## 目标

把对后续规划仍然有价值的信息，持续沉淀为长期上下文，而不是只停留在当前线程里。

优先沉淀：

- 稳定事实背景
- 当前现状变化
- 已收敛的阶段目标
- 当前工作方向变化
- 明确仍待决的问题
- 应长期引用的资料入口

通常不要沉淀：

- 一次性灵感
- 尚未收敛的方案细节
- 没有后续价值的临时讨论
- 纯实现层小修小补

## 什么时候需要更新

当出现以下任一情况时，应该考虑同步长期上下文：

- 新增了稳定背景事实
- 修正了之前的背景判断
- 当前阶段目标或范围发生了变化
- 当前工作方向收敛了新结论
- 出现了会影响后续规划的开放问题
- 新增了需要长期引用的文档、附件、代码入口

## 写入前的硬约束

更新上下文前，必须先给出拟更新摘要，得到用户确认后才能落盘。

如果用户明确说了以下表达，应默认停留在摘要阶段：

- “先汇总看一下”
- “先别落盘”
- “先确认”
- “告诉我会改哪些文件”
- “整理成更新计划”

## 推荐流程

### 1. 提取候选更新项

从当前讨论中提取：

- 新增事实
- 被修正的旧事实
- 当前方向变化
- 新出现的开放问题
- 新增资料入口

### 2. 先做分类

至少区分为：

- 项目总览 / 背景
- 当前现状
- 当前目标与范围
- 当前工作方向
- 待决问题
- 资料索引

如果只收到一条单独背景或一条单独开放问题，也不要直接写文件；仍要先做分类判断。

### 3. 输出拟更新摘要

建议使用这样的结构：

```md
## 拟更新上下文

- 事实背景：
- 当前现状：
- 当前目标与范围：
- 当前工作方向：
- 待决问题：
- 资料索引：

## 计划修改文件

- `docs/context/...`
- `docs/context/...`

## 不写入项

- ...

请确认是否写入。
```

### 4. 确认后再写入

写入时优先遵循：

- 改动最少但语义清晰
- 保持各文档职责边界
- 不在多个文件重复堆砌同一信息
- 如果结构变化，记得同步入口文档

对于背景类信息，额外检查：

- 这是稳定事实，还是暂时判断
- 是新增事实，还是旧事实修正
- 应写入背景/现状，还是只进入待决问题
- 是否会和已有背景表述冲突

### 5. 写后回执

写完后要说明：

- 改了哪些文件
- 每个文件补了什么
- 哪些内容本轮没有写入

如果本轮只停留在摘要阶段，也要说明：

- 哪些内容已归类
- 建议后续写入哪些文件
- 为什么当前未落盘

## 与目录结构治理的边界

本协议只处理“上下文内容如何更新”。

如果问题变成：

- `docs/` 目录如何设计
- `context / designs / plans` 如何分层
- 版本目录如何命名
- `review/`、`comparisons/` 如何归档

那么这已经超出本协议范围，更适合使用 `docs-structure-manager`。
