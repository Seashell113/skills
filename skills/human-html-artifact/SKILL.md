# Human HTML Artifact

这个 skill 的定位是把已有材料做成给人快速理解、探索、评审、复用和决策的 HTML sidecar 视图。它不是“把 Markdown 套一层网页样式”，也不是按固定文档类型套模板；它要先判断源材料中哪些阅读任务是 Markdown 做不好的，再选择 HTML 能明显放大的表达方式。

## 分发定位

对外分发时，应把本 skill 描述为“复杂 Markdown 材料的 HTML sidecar 视图生成器”。不要把它包装成通用 Markdown renderer、博客渲染器或 README 预览工具。适合它的材料通常有评审、决策、下钻、对照、筛选、复制复用或演示任务；普通 Markdown 预览和源文档维护应继续使用 Markdown 工作流。

## 核心原则

- 先判断 HTML 是否能带来真实理解效率增益。若只是维护源文档、普通预览或纯文本 diff，保持 Markdown。
- 默认保留源 Markdown 或源材料不变，生成同名或清晰命名的 sidecar HTML。
- HTML 是交付视图，不是 canonical source。关键事实、结论、风险和来源必须能追溯回源材料。
- 先找阅读瓶颈，再设计页面。不要先问“这是什么文档类型”，而要先问“读者打开它要完成什么任务，Markdown 阻碍在哪里”。
- 充分利用 HTML 相对 Markdown 的优势：空间布局、并排对照、筛选搜索、折叠下钻、局部状态、内联 SVG、复制导出、轻量模拟、图形化关系和角色视角切换。
- 中文技术文档和中文评审语境默认输出中文 UI；若用户指定其他语言，或源材料明显是英文/其他语言，界面语言应跟随用户要求或源材料语言。技术名词、字段、代码、产品名和原文引用按源材料保留。
- 首选单文件、自包含、可离线打开的 HTML。不要依赖 CDN、远程字体、远程图片、外部 JS 或构建步骤。
- 默认视觉基线使用 `references/DESIGN.md` 的 Vercel-like developer-platform 风格：近白画布、ink 黑白主对比、细 hairline、Geist/Inter 风格 sans、mono 技术标签、克制圆角和轻阴影。用户明确指定视觉系统时优先服从用户指定；不要在 skill 中叠加过多整体视觉规则。

## 触发与不触发

适合使用：

- 用户要求把方案、报告、调研笔记、review、周报、状态页、设计说明或代码审查材料做成 HTML。
- 原文存在多阶段、多风险、多角色、多字段、多模块、多方案、多证据、多 diff 或多决策项，线性阅读成本高。
- 用户想让读者更快判断“该不该推进 / 怎么改 / 先看哪里 / 风险在哪里 / 证据是什么 / 能复制什么继续工作”。
- 用户没有详细规划交互，只说“帮我转成 HTML”“做成好读的页面”“给评审会看”。这正是本 skill 要补足的信息设计部分。

不适合使用：

- 用户只想修 README、维护 Markdown 结构、做纯文本 diff，或明确不要 HTML。
- 用户只是要普通 Markdown 预览，没有额外阅读、评审、演示、下钻、对比或复用目标。
- 任务本质是长期维护的网站、复杂前端应用、登录系统或多页面产品。
- 用户要求严格保留原文逐行结构，不允许重组信息。

遇到边界不清时，先用一句话说明判断：如果 HTML 不明显改善理解效率，就保持 Markdown；如果存在对照、筛选、下钻、图形化或复制复用需求，再生成 HTML。

## 复杂度与收益判断

HTML artifact 的收益不只来自文档长度，而来自“信息形状”。遇到下列信号，应主动做 affordance-first 设计：

- 阶段、时间、依赖、迁移路径、状态流转分散在文本中。
- 风险、优先级、影响面、阻塞项、待决策项需要横向比较。
- 字段、配置、token、模块、仓库、API、指标需要筛选、查找或对照。
- PR/MR review 中有文件路径、行号、diff、测试缺口和开放问题。
- 当前/未来、before/after、方案 A/B、保留/移除等关系容易被线性文字混淆。
- 源材料有大量证据、摘录、命令、JSON、代码片段，需要下钻和复制。
- 读者角色不同：管理者看结论，评审人看风险和证据，执行者看命令和行动项。

短文也可以生成 HTML，但必须轻量。不要为了“显得像 artifact”给短状态报告强塞复杂仪表盘、编辑器或多层交互。

## 工作流程

1. **确认源与目标**
   - 找到源材料路径或内联内容。
   - 明确目标读者、使用场景和交付文件名；若用户没说，按材料内容推断并在 `user_notes.md` 记录。
   - 如果用户指定视觉系统，优先使用用户指定；否则使用 `references/DESIGN.md` / Vercel-like 默认视觉基线。
   - 默认不修改源文件；若要改源 Markdown，必须是用户明确要求。

2. **先写 Affordance Plan**
   - 在生成前先用 5-8 行内部规划：这份材料中 Markdown 的阅读瓶颈是什么，读者需要完成哪些任务，HTML 要放大哪些能力。
   - 不要把 artifact 主类型当成唯一骨架。文档类型只作为弱信号；真正决定页面结构的是信息形状和读者任务。
   - 对复杂材料，至少选择 2-4 个有明确收益的 HTML affordance；对短材料，选择 0-2 个轻量 affordance。

3. **按信息形状选择 HTML affordance**
   - 阶段 / 时间 / 依赖：使用 timeline、phase rail、dependency map、状态流转图。
   - 风险 / 优先级 / 影响面：使用 risk matrix、severity lane、heatmap、阻塞项看板。
   - 字段 / 配置 / token / 模块：使用可筛选矩阵、色板、组件状态对照表、模块地图、字段职责卡片。
   - diff / 代码 / 命令 / JSON：使用代码证据区、带注释 diff、文件导览、复制按钮、修复摘要导出。
   - 当前 / 未来 / before-after / 方案对比：使用并排对照、tab 切换、差异矩阵、迁移边界图。
   - 长文证据 / 摘录 / 会议记录：使用 summary-to-evidence 下钻、原文折叠附录、来源索引。
   - 决策任务：使用 decision board、open questions、推进条件、合并条件、评审 checklist。
   - 多角色阅读：使用“决策者 / 评审人 / 执行者”入口或视角筛选。
   - 需要继续工作：提供复制为 Markdown、复制命令、复制 diff、复制 JSON、复制摘要或导出当前筛选结果。

4. **组织页面结构**
   - 首屏必须回答：这是什么、读者现在该关注什么、最大风险或最重要结论是什么、下一步是什么。
   - 主体按任务路径组织，而不是按源 Markdown 章节顺序组织。
   - 完整原文或关键摘录可放在末尾附录或折叠证据区；不要让附录成为主阅读路径。
   - 对缺失信息标注“未提供”或“需确认”，不要补造。
   - 同一页面中交互控件要少而清楚。控件服务理解或复用，不为炫技存在。

5. **生成 HTML**
   - 输出一个 `.html` 文件，CSS 和可选 JS 全部内联。
   - 使用语义化结构：`header`、`main`、`section`、`nav`、`table`、`details`、`aside`、`figure` 等。
   - 必要时使用本地 vanilla JS 做复制、筛选、tabs、展开/收起、轻量预览、导出等交互。
   - 交互必须有静态可读 fallback。关闭 JS 后，摘要、主要内容、结论、风险、证据和行动项仍然可读。
   - 代码块、diff、命令、JSON、文件路径、配置片段、可回填给 agent 的摘要或导出结果，默认提供“复制”按钮或等价可复制控件。
   - 复制逻辑使用 `navigator.clipboard.writeText`，并提供隐藏 `textarea` + `document.execCommand('copy')` fallback；不要为了复制引入外部库。
   - 内联 SVG、图表源、可导出的图形片段如果要求可复制，复制内容必须是完整 SVG/XML 标记。优先对目标 `svg` 使用 `new XMLSerializer().serializeToString(svgNode)`。
   - 页面 UI 语言保持一致。中文任务中，除源材料标题、代码 token 和固定术语外，不要出现无必要英文按钮、导航、状态或区块标题；非中文任务跟随用户要求或源材料语言。
   - 做响应式布局、打印样式和基础可访问性。移动端不依赖固定宽度；打印时隐藏纯交互控件但保留核心内容。

6. **交付前自检**
   - 对照 `references/quality-rubric.md` 检查自包含、事实忠实、HTML lift、响应式、打印、可访问性和本地交互边界。
   - 检查是否真正利用了 HTML：如果页面去掉 CSS/JS 后只剩原 Markdown 顺序，说明没有完成任务。
   - 检查是否使用用户指定视觉系统；若未指定，检查是否符合 `references/DESIGN.md` / Vercel-like 默认基线。
   - 做语言快扫：中文任务中，导航、按钮、筛选、状态、空态、错误、导出、提示和区块标题默认中文；非中文任务保持同一种界面语言。
   - 对复制交互做最小 smoke test：代码/命令/JSON/path 能复制纯文本；SVG 复制得到以 `<svg` 开头的完整标记；失败时页面仍可选择原文。
   - 若环境适合并且 artifact 复杂，可用浏览器打开检查布局；否则至少静态检查 HTML 中没有 `http://`、`https://`、`cdn` 外链。
   - 最终回复说明：源文件未替代、HTML 文件路径、核心 HTML affordance、任何刻意保留为 Markdown 的理由。

## 输出约定

生成 HTML 时，默认交付：

- `source-name.html` 或用户指定路径。
- 必要时附 `user_notes.md`，记录 artifact 判断、选择的 HTML affordance、无法验证的事实、假设和需要人工确认的点。

如果判断不应 HTML 化，交付应包含：

- 保留 Markdown 的理由。
- 可选的 Markdown 改写建议或轻量预览方案。
- 不要为了“用了 skill”而强行创建 HTML。
