# 大前端统一埋点采集与分析体系 v1.6 阶段性方案

> 文档状态：正式评审前版本  
> 背景：v1.6 替代 v1.5，不再机械沿用“全量入数仓”的长期方案  
> 评审目标：确认当前阶段是否采用 SLS 作为行为数据承接层，并明确未来入仓边界

## 1. Summary

本版方案建议当前阶段采用：

- 行为数据：进入 `SLS + SLS 仪表盘`。
- 业务数据：继续走现有数仓 / `QuickBI`。
- 少量跨域联查：按专题处理，不默认建设正式主题层。
- 未来正式入仓：等统一契约稳定后再启动。

本版不是 v1.5 的轻微改写，而是重新判断三件事：

1. 哪些字段值得进入主契约。
2. 哪些字段只适合当前阶段做 SLS 展示或调试。
3. 当前阶段与未来数仓分层是否被混写。

## 2. 当前阶段主链路

当前只描述真实启用链路：

```text
多端 SDK
  -> SLS 原始行为事件
  -> SLS 查询 / SLS 仪表盘 / SLS 告警
```

不把 `DWD / DWS / QuickBI` 写进当前主链路。它们属于未来正式入仓目标，而不是当前默认启用的生产链路。

## 3. 未来入仓目标链路

未来链路单独描述：

```text
稳定版本切点
  -> ODS 原始行为明细层
  -> DWD 标准化行为明细层
  -> DWS 行为主题层
  -> QuickBI 数据集与看板
```

重要边界：

- `DWD` 只承接稳定公共字段和被准入的分析维度。
- `DWS` 只沉淀稳定、可复用、需要进入正式经营分析的行为主题。
- 当前 SLS 看板逻辑不能直接平移成未来 DWS 主题。

## 4. 字段职责分层

### 4.1 稳定主契约字段

这些字段代表稳定公共语义，默认进入主契约候选：

| 字段 | 含义 | 评审关注 |
|---|---|---|
| `event_id` | 事件唯一标识 | 必须全局唯一 |
| `event_key` | 事件编码 | 需台账治理 |
| `event_type` | 事件类型 | 需稳定枚举 |
| `event_time` | 事件时间 | 时间源需统一 |
| `schema_version` | 契约版本 | 兼容策略需明确 |
| `sdk_version` | SDK 版本 | 用于排障 |
| `biz_line` | 业务线 | 进入固定分析 |
| `app_id` | 应用标识 | 多端聚合依据 |
| `topic` | 主题 | 需避免滥用 |
| `platform` | 平台 | Web / 小程序 / App |
| `user_id` | 用户标识 | 权限与脱敏要求 |
| `tenant_id` | 租户 | B 端分析核心维度 |
| `shop_id` | 门店 | 门店分析核心维度 |
| `session_id` | 会话 | 当前技术链路字段 |
| `duration_ms` | 时长 | 固定分析便利槽 |
| `trace_biz_id` | 业务追踪 | 跨域排障 |
| `traffic_source_type` | 流量来源类型 | 需要稳定枚举 |

### 4.2 展示辅助字段

| 字段 | 当前用途 | 限制 |
|---|---|---|
| `event_name` | SLS 看板展示可读名称 | 不作为语义真相 |
| `view_name` | 页面展示名 | 权威名称来自台账 |

### 4.3 分析维度字段

| 字段 | 当前用途 | 未来入仓判断 |
|---|---|---|
| `device_os` | 按系统切片 | 按需求选择性入仓 |
| `device_os_version` | 系统版本细分 | 需要控制基数 |
| `device_model` | 设备型号分析 | 可能高基数 |
| `device_screen_width` | 屏幕宽度 | 当前看板可用 |
| `device_screen_height` | 屏幕高度 | 当前看板可用 |
| `device_network` | 网络类型 | 体验分析 |

### 4.4 观察调试字段

这些字段保留在 raw event，用于排障、专题观察和临时分析，不进入默认固定看板：

- `page_url`
- `page_path`
- `page_params`
- `device_user_agent`
- `traffic_source_detail`

### 4.5 受控辅助字段

`user_phone` 调整为受控辅助字段：

- 可在 raw event 中按需保留。
- 不作为默认看板维度。
- 不作为默认标准化明细字段。
- 不列入稳定主契约。
- 仅在明确业务必要性和权限审批下用于专题查询、脱敏展示或导出。

## 5. 从主方案移除的字段

小程序页面栈 `page_stack / pagePaths` 从主方案移除：

- 不进入统一埋点主契约。
- 不作为默认观察字段。
- 若未来确有专项调试价值，再以专项调试日志或 SDK 调试能力设计。

## 6. 治理与台账约束

台账至少新增字段：

| 台账字段 | 说明 |
|---|---|
| 稳定性分级 | 稳定字段 / 分析维度 / 观察调试 / 受控辅助 |
| 是否候选入仓 | 是否未来进入 ODS / DWD / DWS |
| 消费归属 | SLS 看板、专题、QuickBI、排障 |
| 指标引用 | 是否被正式指标使用 |
| 看板引用 | 是否被固定看板使用 |

新增扩展字段不能再遵循“能采就能查、能查就能进看板”。每个字段必须先明确归类和用途。

## 7. 退出标准与迁移启动标准

### 7.1 SLS 退出评估标准

出现以下信号时，启动正式入仓评估：

- 跨域联查从零散专题变成常态化需求。
- 行为指标需要稳定进入经营分析或正式业务报表。
- SLS 看板和口径维护明显重复、漂移或难以治理。
- 运营、产品持续要求复用化行为主题数据。
- 数仓侧具备承接稳定行为主题的意愿和能力。

### 7.2 迁移启动标准

迁移不是按日期自动触发，而是满足条件后启动：

- 统一契约经过观察期后明显稳定。
- 台账能识别稳定事件、稳定指标和候选入仓范围。
- 数仓侧明确 ODS / DWD / DWS / 验证链路责任。
- 业务侧出现需要正式复用的行为主题，而不只是临时查询和固定看板。

## 8. 评审需要防止的误读

1. 不要把 SLS 理解成长期统一分析平台。
2. 不要把未来数仓分层写成当前已经启用的链路。
3. 不要把所有顶层字段同等看待。
4. 不要把 `event_name / view_name` 当成语义真相。
5. 不要把 `user_phone` 当成默认维度。
6. 不要让 `event_data` 变成没有治理的万能字段。

## 9. 评审阅读难点

线性 Markdown 很难同时看清：

- 当前链路和未来链路的分界。
- 字段分层与入仓资格的关系。
- SLS 退出标准与迁移启动标准的差异。
- 哪些字段保留、降级、移除、受控。
