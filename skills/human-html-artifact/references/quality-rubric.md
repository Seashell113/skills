# Quality Rubric

用这份清单做交付前检查。目标不是追求炫技，也不是把每类 Markdown 套成固定 HTML 模板，而是确认 HTML artifact 是否真正利用了 HTML 相对 Markdown 的表达优势，帮助人更快阅读、探索、评审、复用和决策。

## 必过项

1. **源材料地位清晰**
   - 源 Markdown 或输入材料没有被隐式替代。
   - HTML 中说明来源、更新时间或输入文件。
   - 最终回复说明 HTML 是 sidecar 视图。

2. **事实忠实**
   - 关键事实、风险、结论能在源材料中找到依据。
   - 未提供的信息被标注为待确认，而不是补造。
   - 没有为了美化页面新增未经验证的指标、时间或结论。

3. **自包含**
   - HTML 不依赖 CDN、远程字体、外部 JS、外部 CSS 或远程图片。
   - CSS 和必要 JS 内联。
   - 离线打开也能阅读核心内容。

4. **HTML lift**
   - 对复杂材料，页面至少提供 2 个 Markdown 难以做到、且服务读者任务的表达增强，例如并排对照、筛选矩阵、风险泳道、阶段图、带注释 diff、摘要到证据下钻、内联 SVG、复制导出、轻量状态预览。
   - 每个增强都应能解释它解决了哪个阅读或复用问题。
   - 如果去掉 CSS/JS 后只剩原 Markdown 顺序和少量卡片，说明 HTML lift 不足。
   - 短材料可以轻量处理，但不能用复杂控件伪装增益。

5. **信息架构**
   - 首屏提供摘要、结论、风险、下一步或读者任务入口，不让读者先读长背景。
   - 主体按读者任务和信息形状组织，而不是机械复制原 Markdown 层级。
   - 长复杂文档要有多层阅读路径：总览、对照/矩阵/图表、下钻证据、原文追溯。
   - 文档类型只是弱信号；不要求同类文档每次固定同一章节顺序。
   - 主阅读区不能只是把 Markdown 按原顺序包进样式模板。

6. **视觉基线**
   - 默认使用 `references/DESIGN.md` 的 Vercel-like developer-platform 风格；用户明确指定其他视觉系统时，以用户指定为准。
   - 检查关键气质即可：近白画布、ink 主对比、细 hairline、Geist/Inter 风格 sans、mono 技术标签、克制圆角和轻阴影。
   - 视觉约束服务于可读性和一致性，不压过具体内容和 HTML affordance。
   - 技术文档 artifact 默认不需要 gradient 装饰。不要把 `references/DESIGN.md` 的 mesh gradient 机械套到普通评审页、报告页或状态页。

7. **响应式**
   - 移动端不依赖固定宽度。
   - 长表格、代码块、路径可以滚动或换行，不撑破页面。
   - 标题、按钮、标签不会互相遮挡。

8. **打印可用**
   - 有 `@media print` 样式。
   - 打印时保留标题、来源、摘要、主要内容和关键表格。
   - 纯交互控件可隐藏，但内容不能随控件一起消失。

9. **可访问性**
   - 使用语义化 HTML。
   - 颜色对比足够，风险等级有文字标签。
   - 交互控件有文本或 `aria-label`，键盘可操作。
   - hover 信息必须能通过键盘 focus 或正文文本获得，不能只对鼠标用户可见。

10. **自包含与本地交互边界**
   - 允许使用内联 vanilla JS 做复制、筛选、tabs、展开/收起、轻量预览和导出，但核心内容不能依赖 JS 才可读。
   - 不发网络请求，不加入分析脚本、远程资源或自动外传内容的行为。
   - 不加载第三方库、远程字体、远程图片或外部样式。

11. **语言一致性**
   - 中文任务中，界面文案、按钮、状态标签、风险说明和提示默认中文；用户明确指定其他语言，或源材料明显是英文/其他语言时，按用户要求或源材料语言保持一致。
   - 英文只保留必要技术名词、字段名、代码标识、产品名和原文引用。
   - 中文任务中，导航、区块标题、表格表头、筛选控件、复制按钮、导出按钮和反馈不使用裸英文 UI 标签，如 `Copy`、`Jump link`、`Feature Explainer`、`Status strip`、`Prompt tuner`、`glossary`、`prompt pack`、`feature flag groups`。
   - 必须保留英文术语时，中文先行、英文括注或放在技术标签中。

12. **交互价值**
   - 交互控件要服务读者任务，不为炫技增加负担。
   - 关闭 JS 后仍能看到完整结论、风险、证据和行动项。
   - 代码块、diff、命令、JSON、文件路径、配置片段和导出结果按复用需要提供复制按钮或等价控件；复制失败或关闭 JS 时文本仍可选择。
   - 筛选、搜索或 tabs 应提供重置、当前状态或数量反馈，避免读者不知道页面隐藏了什么。
   - 如果提供 SVG 复制，复制结果必须是完整 `<svg>` 标记；复制图中文字或空字符串视为失败。

13. **Review / Diff 上下文**
   - PR/MR review 或代码审查 artifact 必须保留关键代码上下文：文件路径、行号、相关 diff 或代码片段。
   - 每段 diff 旁边要有中文影响说明和建议，而不是只展示代码。
   - 源材料没有 diff 时，要明确标注“未提供 diff”，不能自行编造代码。

## 加分项

- 在 `user_notes.md` 中简短记录 Affordance Plan：Markdown 瓶颈、读者任务、选择的 HTML 放大点。
- 有清晰的 review checklist、decision board 或合并 / 推进条件。
- 对事实、判断、建议做了视觉区分。
- 对长内容提供折叠、筛选或目录，但关闭 JS 后仍可读。
- 关键代码块、路径、命令和摘要有一键复制，并有“已复制”反馈。
- 编辑器或状态报告能导出 / 复制为 Markdown、JSON、diff 或用户指定格式。
- 图表有文字解释和打印 fallback。
- 风险、行动项、负责人、时间点能被快速扫读。
- 对字段、阶段、风险和代码证据提供 hover/focus 解释或下钻。
- 对多角色材料提供“给决策者看 / 给评审人看 / 给执行者看”的任务入口。

## 常见失败

- 只是把 Markdown 包进一个模板，没有改善阅读路径。
- 主体仍按原 Markdown 线性输出，只是加了颜色和卡片。
- 过早套用“技术方案 / 评审包 / briefing / 状态页”固定骨架，忽略材料真正的信息形状。
- 页面更长、更贵、更复杂，但没有比 baseline 更快回答读者问题。
- 中文任务里大量使用英文 UI 标签，影响阅读顺畅度。
   - 默认任务没有使用 `references/DESIGN.md` / Vercel-like 基线，或用户指定视觉系统时没有服从用户指定。
   - 在 `references/DESIGN.md` 之外继续叠加大量无关视觉规则，导致页面和任务内容过耦合。
   - 把 `references/DESIGN.md` 的 mesh gradient 当成普通文档默认装饰，叠多层渐变或随机渐变卡片。
- 用大量装饰、渐变或复杂布局压过内容。
- 有代码块、命令或路径但没有复制按钮，读者仍要手动选中。
- SVG 复制按钮实际复制的是 `innerText`，导致只能拿到图中文字，无法复用图形。
- 有筛选控件但没有重置、数量反馈或无 JS fallback。
- 引入 Tailwind、Bootstrap、Mermaid、Chart.js 等 CDN，导致离线不可用。
- 把 HTML 当成新事实源，修改或覆盖了原 Markdown。
- 交互筛选后内容不可打印，或关闭 JS 后页面空白。
- 对“不需要 HTML”的 README 或源文档维护任务也强行生成网页。
- code review 页面只列 finding，不展示关键代码片段或 diff 上下文。

## 快速静态检查

交付前至少检查：

```bash
rg -n "https?://|cdn|unpkg|jsdelivr|googleapis|analytics" output.html
```

如果命中，确认是不是源材料里的普通文本引用。若是资源引用，应改为内联或移除。
