# HTML Affordance Patterns

这份参考不是“文档类型模板库”，而是 HTML affordance 模式库。使用时先判断材料的信息形状和读者任务，再选择局部模块。文档类型可以提供线索，但不要让“技术方案 / review / briefing / status”这些标签决定整页结构。

好的 HTML artifact 应该让读者完成 Markdown 很难完成的动作：横向比较、筛选定位、按角色扫读、从摘要下钻到证据、复制可复用片段、查看关系图、理解阶段边界、模拟局部状态或导出下一步工作材料。

中文技术文档和中文评审语境默认输出中文界面和中文提示；用户明确指定其他语言，或源材料明显是英文/其他语言时，界面语言跟随用户要求或源材料语言。技术名词、代码标识、字段名、产品名和原文引用按源材料保留。

## 视觉基线

默认使用 `references/DESIGN.md` 的 Vercel-like 基线，可内置到生成结果中；用户明确指定其他视觉系统时，以用户指定为准。

- **色彩**：近白画布、ink 黑白主对比、细 hairline；语义色只小面积表达状态、风险、阶段和选择。
- **字体**：Geist/Inter 风格 sans 做正文和按钮，mono 字体做技术标签、代码、路径和计数；不加载远程字体。
- **版面**：页面最大宽度约 1120-1400px；首屏保留标题、来源、核心判断和关键入口；主体用细边框、8-12px 克制圆角和轻阴影。
- **装饰**：普通技术文档 artifact 默认使用平面近白画布，不把 mesh gradient 当成必要装饰。只有用户要求品牌化 hero、视觉方向探索或演示页时，才小面积使用更强视觉表达。
- **稳定性**：稳定的是视觉语言和控件习惯，不是每类文档固定章节。页面结构应由信息形状驱动。

## 语言规则

- 中文任务中，导航、按钮、筛选、状态、表格表头、空态、错误、导出、复制反馈和区块标题默认中文。
- 英文术语只在三类场景保留：代码/字段/路径/API 名；产品名或协议名；源材料中的原文引用。保留时提供中文上下文。
- 中文任务中，不要让常规 UI 出现 `Copy`、`Open`、`Jump link`、`Status`、`Owner`、`Risk`、`Feature Explainer`、`Prompt tuner`、`glossary`、`prompt pack`、`feature flag groups` 这类裸英文。
- 如果原文流程列名本身是英文，如 `Now / Next / Later / Cut`，中文界面里使用“现在 / 下一批 / 稍后 / 裁剪（Now / Next / Later / Cut）”或类似双语标签。
- 非中文任务保持界面语言一致，不要混用中文默认按钮和英文正文。

## Affordance Plan 模板

复杂 artifact 生成前先做这个规划。规划不一定写进页面，但应体现在 `user_notes.md` 或最终交付说明中。

```text
读者任务：
- 读者要快速判断什么？
- 读者需要比较、筛选、复制、下钻或演示什么？

Markdown 瓶颈：
- 哪些关系在线性文本里容易混淆？
- 哪些事实分散在多个章节？
- 哪些片段需要被复用？

HTML 放大点：
- 使用哪些 2-4 个 affordance？
- 每个 affordance 解决哪个阅读或复用问题？
```

## 模式 1：摘要到证据下钻

适用于长文、调研、评审稿、事故复盘和多来源摘录。

- 首屏给 3-5 条结论、最大风险、下一步和来源。
- 每条结论旁边提供“查看证据”入口，跳到对应摘录、diff、指标或原文段落。
- 原文附录保留追溯，但不作为主阅读路径。
- 证据区使用 `details/summary`，关闭 JS 后仍完整可读。

适合解决：长文先读哪里、结论来自哪里、哪些内容需要核验。

## 模式 2：风险矩阵和风险泳道

适用于技术方案、PR review、项目状态、上线决策和安全/稳定性评估。

- 用严重度、影响面、触发信号、缓解措施四类信息组织风险。
- 风险先按阻塞/非阻塞或 P0/P1/P2 排序，再允许按文件、模块、阶段、负责人或状态筛选。
- 每条风险关联证据：源段落、diff、指标、日志、命令或开放问题。
- 提供“只看阻塞项”“复制风险摘要”“复制合并条件”等控件。

适合解决：风险淹没在正文里、无法判断先处理什么、reviewer 需要快速定位证据。

## 模式 3：阶段、时间线和依赖图

适用于迁移计划、路线图、事故时间线、发布计划和流程改造。

- 顺序关系用 timeline 或 phase rail，依赖关系用 dependency map，不要用长段落描述。
- 每个阶段包含目标、输入、输出、负责人/角色、验证方式、退出条件。
- 对容易误读的边界，用“当前 / 未来”“已启用 / 未启用”“本地 / CI”等标签明确区分。
- 支持阶段筛选或点击阶段下钻到命令、配置、风险和验收标准。

适合解决：阶段边界混淆、当前状态和目标状态混写、执行细节分散。

## 模式 4：对照和差异矩阵

适用于方案 A/B、before/after、当前/未来、保留/移除、旧逻辑/新逻辑。

- 使用并排列、差异矩阵或 tab 切换。不要把多个方案写成连续长章节。
- 比较维度保持一致：成本、风险、复杂度、适用条件、迁移代价、回滚方式。
- 对 before/after，明确哪些行为不变、哪些行为改变、哪些仍待确认。
- 对代码或配置差异，提供可复制片段。

适合解决：读者需要横向比较而不是顺序阅读。

## 模式 5：字段、配置、token 和模块矩阵

适用于字段治理、埋点方案、设计系统、配置文档、API 参数和模块地图。

- 将字段/配置/token/module 统一成矩阵：名称、类别、职责、来源、状态、风险、是否保留、是否可复用。
- 提供筛选、搜索、计数反馈和重置。
- 颜色色板、间距尺、字号阶梯、组件状态对照表可以直接渲染，不只写文字说明。
- 对重要字段或 token 提供复制名称和值的按钮。

适合解决：字段/配置散落、多维属性难扫读、设计 token 无法视觉比较。

## 模式 6：代码证据和 annotated diff

适用于 PR/MR review、代码审查、风险复审和实现方案说明。

- 展示最小必要 diff 或代码片段，保留 `-` / `+` 视觉区分。
- 每段 diff 旁边说明影响、触发条件、建议修复和测试缺口。
- 文件路径、函数名、diff、修复摘要都应可复制。
- 如果源材料没有 diff，不要编造；明确标注“未提供 diff，只能展示文件/行号和文字证据”。

适合解决：只有评论没有上下文、作者不知道怎么改、reviewer 需要判断证据是否成立。

## 模式 7：内联 SVG / CSS 图示

适用于数据链路、架构边界、状态机、流程图、依赖图、概念解释和简单可视化。

- 优先使用内联 SVG、HTML/CSS 图或静态表格，保持单文件离线可读。
- 每张图都有标题、图例、读图说明和文本 fallback。
- SVG 如需复制，复制完整 `<svg ...>...</svg>` 标记，而不是图中文字。
- 不依赖 Mermaid、Chart.js、D3、外部图片或远程字体。

适合解决：关系只能靠长段文字描述、当前/未来链路容易混淆、概念需要直观看到。

## 模式 8：角色视角和任务入口

适用于同一材料服务多个读者：管理者、评审人、执行者、作者、设计系统维护者、业务接入方。

- 首屏提供 2-4 个角色入口或任务入口。
- 每个入口不是换一套页面，而是跳到该角色最关心的结论、风险、证据或行动项。
- 不要让角色入口隐藏核心内容；关闭 JS 或打印时仍按完整顺序展示。
- 角色入口适合和摘要、风险、行动项、证据区联动。

适合解决：一个文档同时给多人看，所有人都被迫读完整线性文档。

## 模式 9：轻量模拟、调参和状态预览

适用于交互方案、动效参数、提示词模板、配置开关、ticket 排序和功能开关编辑。

- 只在源材料确实包含可调参数、模板变量、排序、开关或状态组合时使用。
- 控件要少而明确，并提供“重置”“复制当前配置 / diff / Markdown / JSON”。
- 关闭 JS 后，默认状态、所有可选项和导出格式仍可读。
- 不使用网络请求、localStorage 或外部库。

适合解决：读者需要试参数、看状态组合、产出下一步可用配置，而不是只阅读说明。

## 复制控件实现要求

- 按钮文案默认中文，如“复制代码”“复制路径”“复制命令”“复制 diff”“复制为 Markdown”。
- 按钮放在代码块或片段的 header 行，不要遮挡代码内容；移动端仍可点击。
- 成功反馈默认“已复制”，1-2 秒后恢复原文案；失败时保持文本可选，并可提示“请手动选择复制”。
- 复制内容应是纯文本，不包含 HTML 标签、行号 UI 或视觉标记；diff 可保留 `+` / `-`。
- SVG 复制是例外：复制内容应是完整 `<svg ...>...</svg>` XML 标记。实现时用 `XMLSerializer().serializeToString(svgNode)`；必要时补上 `xmlns="http://www.w3.org/2000/svg"`。
- 复制按钮可以用 `data-copy` 显式给出纯文本，也可以用 `data-copy-target` 指向 `pre`、`code`、`textarea`、`input` 或 `svg`。取值顺序建议是：SVG 序列化 -> 表单 `value` -> `data-copy` -> `textContent`。

## 文档类型只是弱信号

下面只是起点，不是固定模板：

- 技术方案常见 affordance：阶段时间线、风险矩阵、决策板、命令下钻、当前/未来边界图。
- PR review 常见 affordance：风险泳道、带注释 diff、文件导览、修复摘要复制、合并条件。
- 调研材料常见 affordance：摘要到证据、方案对照、字段矩阵、可信度/待确认筛选。
- 状态报告常见 affordance：状态条、阻塞项看板、指标证据、行动项复制、周会摘要导出。
- 设计系统材料常见 affordance：色板、字号阶梯、间距尺、组件状态对照表、token 复制。

如果材料的信息形状与文档类型标签冲突，优先服从信息形状。
